# 專案簡介（HR / 招募人員適用）

## 一句話摘要

這是一個以洗錢防制（AML）紅旗辨識為應用場景、可實際執行的 AI 工程 Portfolio 示範專案。

---

## 這個 Demo 模擬什麼問題

洗錢防制調查員在審查交易時，需要快速判斷某筆行為是否符合已知的洗錢紅旗指標。
本專案模擬這個查詢流程：根據用戶提問，從知識庫中檢索相關規則與案例，判斷是否
存在紅旗，並附上來源引用。

---

## 我做了什麼

- 以 FastAPI 建立一套可執行的 RAG 查詢 API（健康檢查、查詢、知識庫來源三個端點）
- 實作 BM25、Dense FAISS、Hybrid RRF 三種檢索模式，並能在不可用時自動降級與回報
- 設計 Pre-LLM Gate：在進入檢索前先判斷問題是否在範圍內，範圍外直接拒答
- 支援 Mock、Groq、Gemini/Gemma、Ollama 多種生成路徑；Mock 為預設，不需 API Key
- 實作選擇性啟用的結構化多輪對話記憶，記錄活躍情境、識別到的紅旗與來源引用
- 設計確定性 Intent Routing，依規則判斷每一輪的處理方式（新查詢、從記憶回覆、要求澄清、或拒答）
- 撰寫 121 個自動化測試及多組評估腳本（API smoke、跨查詢一致性、多輪對話評估）

---

## 這展示了哪些 AI 工程能力

- RAG pipeline 設計（檢索、Gate、生成、引用）
- 多種檢索策略的整合與降級處理
- 結構化多輪對話狀態管理
- 確定性路由設計（不依賴 LLM 做路由決策）
- 測試驅動開發：API contract tests + 多輪評估腳本
- 可觀測性：debug 欄位、fallback 原因、引用來源
- 多 LLM 後端整合（含本地 Ollama 路徑）

---

## 為什麼與 RAG / AI 平台 / 內部知識助理職位相關

本專案直接示範 RAG 系統的核心工程挑戰：

- 如何設計讓系統只在有根據時才回答（Gate + 引用）
- 如何處理跨語言文件的混合檢索
- 如何在多輪對話中維持結構化狀態而不膨脹
- 如何讓系統在外部依賴失敗時仍能有意義地降級

這些問題在 AI 平台工程、內部知識助理、法務合規輔助工具等場景中普遍存在。

---

## 這個專案不是什麼

- 不是真正的 AML 合規系統，也不提供法律意見
- 不是生產環境部署的服務（不含認證、授權、稽核日誌）
- 不是 LLM 品質或模型選型的比較 benchmark
- 知識庫為示範用小型語料，不代表真實的 AML 覆蓋率
